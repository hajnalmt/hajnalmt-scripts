# Arm interview postmortem: valid number

Date: 2026-07-10
Status: initial notes, to be continued

## What happened

The interview included a traditional live-coding task: implement an `is_valid_number` function that receives a string argument and returns whether the string is a valid number.

A function skeleton was provided. I started writing a `for` loop, but I froze on basic implementation details and did not get to a clean solution.

## What I failed on

- I did not think of using a regular expression during the interview.
- I started writing a manual character-by-character loop.
- I got stuck on basic syntax for string comparison / character checks.
- The interviewer helped me with syntax.
- I still do not remember the exact syntax afterward, which is a useful signal: my mind forgets easily googleable syntax under pressure.

## Important interpretation

This was not a failure to understand software engineering.

This was a failure of live, memory-based, traditional coding performance under interview pressure.

The gap exposed here is not architectural judgment, code review ability, MLOps ability, or senior engineering taste. The exposed gap is:

- small-function implementation fluency,
- syntax recall under stress,
- interview-pattern recognition,
- and nervous-system tolerance for artificial live-coding tasks.

That is painful, but it is trainable.

## Why it felt bad

The task was simple enough that failing it felt humiliating.

The frustrating part is that, in real work, syntax like this is normally searched, checked in docs, generated by an assistant, or recovered quickly from existing code. Even ten years ago, googling syntax was normal.

The interview format made easily recoverable details feel like a measure of competence.

## What this task was probably testing

Whether I could:

- clarify the accepted number grammar,
- list valid and invalid examples,
- choose a parsing approach,
- implement a small validator,
- handle edge cases,
- and stay calm while doing it live.

The task was not really about the exact function. It was about the interview performance loop.

## Missing recovery pattern

A better response would have been:

1. Pause and clarify the accepted grammar.
2. Write examples before code.
3. List edge cases.
4. Choose either regex or a manual parser.
5. Implement a minimal solution slowly.
6. Test aloud against the examples.

For this problem, the examples should include:

| Input | Expected |
| --- | --- |
| `"0"` | true |
| `" 0.1 "` | true |
| `"abc"` | false |
| `"1 a"` | false |
| `"2e10"` | true |

| `"1e"` | false |
| `"e3"` | false |
| `"."` | false |
| `".1"` | true |
| `"1."` | true |
| `"+.8"` | true |

## Concrete mistake log

Problem: valid number parser

Forgot / missed:

- Regex did not come to mind.
- Python string comparison / character-check syntax was not automatic under pressure.
- I started coding before pinning down the grammar and examples.
- I did not have a practiced fallback sequence for this kind of validation problem.

Pattern:

- Validation grammar with optional sign, decimal component, and exponent.
- Could be solved with regex or with a finite-state/manual scan.

Recovery habit:

- Do not rely on remembering syntax perfectly.
- First write examples and edge cases.
- Then implement the smallest version.
- If syntax disappears, say calmly: “I’d normally check exact syntax, but I’ll write the intended logic and correct syntax as I go.”
